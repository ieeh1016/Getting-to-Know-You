PORTABLE AI DEVELOPMENT METHOD

하나의 MD 파일로 어디서든 같은 AI 개발 프로세스를 시작하는 제안

지금 만든 SDD/TDD, context routing, trace ID, verification handoff 구조를 프로젝트마다 다시 설명하지 않고, AI_APP_DEV_PROCESS.md 하나로 복사해 새 앱 개발을 시작하는 방식이다. 이 파일은 코드가 아니라 개발 운영체제의 씨앗이다.

추천 방향

단일 MD는 source of truth가 아니라 bootstrap kernel로 둔다

하나의 MD 파일 안에 모든 spec을 영원히 넣으면 곧 비대해진다. 대신 새 프로젝트 초기에 필요한 문서 구조, 규칙, 템플릿, AI 명령을 생성하는 시작점으로 쓰는 것이 가장 안정적이다.

핵심 가치

앱이 달라도 개발 습관은 복제된다

기능은 프로젝트마다 다르지만, spec 먼저, test 먼저, 작은 구현, focused verification, context 절약 handoff는 반복 가능한 개발 방법론이다.

주의점

한 파일에 모든 내용을 고정하면 다시 토큰이 커진다

MD 하나는 길고 자세한 백과사전이 아니라, 새 repo에 필요한 문서들을 분화시키는 압축된 설계도여야 한다.

ARCHITECTURE

MD 하나가 새 프로젝트에 만들어내는 구조

추천안은 AI_APP_DEV_PROCESS.md를 모든 프로젝트에 복사한 뒤, 첫 AI agent에게 이 파일을 읽고 프로젝트에 맞는 운영 문서를 생성하게 하는 방식이다.

Bootstrap Kernel

AI_APP_DEV_PROCESS.md의 역할

새 프로젝트에 들어가는 단일 파일이다. AI에게 개발 철학, 산출 문서, 작업 순서, 테스트 계약, context routing, handoff 규칙을 알려준다.

ONE FILE TO START AI_APP_DEV_PROCESS.md

docs/spec.md 제품 기준 문서의 진입점
docs/test_plan.md 검증 의도와 regression map
docs/ai_context_map.md 작업별 문서 routing
docs/spec_trace.md spec과 test의 빠른 lookup
docs/design/ UI/UX 합의용 HTML proposal
Boot Flow

새 앱을 시작할 때의 흐름

이 흐름을 따르면 프로젝트마다 같은 기준으로 문서와 테스트가 생기고, AI가 전체 파일을 매번 읽는 습관도 줄어든다.

Step 1
새 repo 루트에 AI_APP_DEV_PROCESS.md를 복사한다.
Step 2
AI에게 “이 방법론으로 프로젝트 운영 문서를 scaffold해줘”라고 지시한다.
Step 3
앱의 product brief를 바탕으로 docs/spec.md와 feature spec seed를 만든다.
Step 4
초기 test plan, context map, spec trace, verification gate를 만든다.
Step 5
첫 feature부터 spec -> failing test -> implementation -> verification 순서로 진행한다.
CONTENTS

단일 MD 안에는 무엇을 넣어야 하나

이 파일은 모든 프로젝트의 세부 spec을 담는 문서가 아니다. 대신 새 프로젝트가 스스로 spec과 test 체계를 갖추도록 만드는 생성 규칙을 담는다.

1 철학과 원칙

SDD/TDD, small change, explicit data write, focused verification, user change 보호 같은 기본 원칙.

2 생성할 문서 목록

AGENTS.md, docs/spec.md, feature spec, test plan, context map, trace table.

3 AI 작업 순서

분류, context routing, spec update, fail-first test, implementation, verification, handoff.

4 템플릿 블록

feature spec 템플릿, test plan 템플릿, spec trace row, handoff packet, failure log summary.

5 프로젝트 어댑터

Flutter, Web, Firebase, Supabase, REST API, local-only 같은 stack별 체크리스트를 선택적으로 붙인다.

6 금지와 예외

test 약화 금지, user change revert 금지, 무분별한 refactor 금지, doc-only 예외 기록.

7 검증 명령 설계

프로젝트별 analyze/test/build 명령과 실패 로그를 줄이는 보고 형식.

8 토큰 절약 규칙

큰 문서와 큰 test file을 열기 전 routing map과 trace table을 검색하는 규칙.

BOUNDARY

PROMPT

새 프로젝트에서 AI에게 줄 첫 요청

단일 MD 파일은 사람이 읽는 문서인 동시에 AI에게 주는 bootstrap instruction이다. 아래 프롬프트를 함께 넣어두면 새 프로젝트 초기 세팅이 흔들리지 않는다.

COMPARISON

이렇게 만들면 와닿고, 이렇게 만들면 무거워진다

단일 MD 전략은 아주 좋다. 다만 그 파일을 “모든 걸 다 담는 문서”로 만들면 다시 토큰을 많이 먹는다. 시작점과 생성기로 쓰는 편이 더 강하다.

위험한 방식

거대한 만능 MD 하나

형태 모든 spec, 모든 테스트 계획, 모든 stack 규칙, 모든 예시를 한 파일에 계속 추가한다.
문제 새 작업마다 AI가 이 긴 파일 전체를 읽게 되고, 시간이 지나면 지금 줄이려던 context 비용이 다시 커진다.
결과 휴대성은 있지만 실무 운영성이 떨어진다.
추천 방식

작은 bootstrap MD 하나

형태 개발 방법론, 산출물 템플릿, AI 부트스트랩 프롬프트, stack adapter 선택 규칙만 담는다.
장점 새 프로젝트마다 필요한 문서가 분리 생성되고, AI는 작업별 routing map으로 필요한 부분만 읽는다.
결과 내 개발 철학은 복제되고, 프로젝트별 문맥은 가볍게 유지된다.

구현 방향 제안

추천 파일명

AI_APP_DEV_PROCESS.md. 이름이 명확하고, 어느 repo에 복사해도 “AI 앱 개발 프로세스의 시작점”임이 바로 보인다.

보관 위치

개인 템플릿 repo 또는 Codex workspace의 reusable docs 폴더에 둔다. 새 앱 시작 시 repo root에 복사한다.

포함할 adapter

기본은 stack-agnostic으로 두고, Flutter/Firebase, React/Supabase, local-first 같은 adapter section을 선택형으로 둔다.

Data write note

특정 앱이 Firebase를 쓰면 Firestore write boundary를 feature spec과 test plan에 분리한다. Firebase가 아니면 DB write/API mutation boundary로 같은 원칙을 적용한다.

첫 구현 범위

다음 단계에서는 실제 AI_APP_DEV_PROCESS.md v0.1을 만들고, 이 repo의 문서들을 일반화한 skeleton과 bootstrap prompt를 넣으면 된다.

성공 기준

새 repo에서 이 MD 하나만 읽힌 AI가 production code를 만들기 전에 운영 문서와 첫 feature spec/test plan을 안정적으로 생성하면 성공이다.
