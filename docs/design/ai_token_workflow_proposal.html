AI DEVELOPMENT PROPOSAL

SDD/TDD를 유지하면서 AI 토큰 비용을 줄이는 운영안

지금의 문서 기반 개발은 품질을 지키는 힘이 있다. 문제는 AI가 매번 너무 많은 기준 문서, 테스트 파일, 검증 로그를 읽으면 작은 변경도 큰 대화가 된다는 점이다. 이 제안서는 품질은 유지하고, AI가 읽는 문맥을 작업 단위로 좁히는 실제 운영 방법을 정리한다.

추천 방향

전체 문서 읽기에서 작업별 컨텍스트 팩으로 전환

SDD/TDD 자체를 줄이지 않는다. 대신 작업 유형마다 반드시 읽을 문서, 검색할 테스트, 열어볼 구현 파일을 작게 정한다. AI에게 필요한 문맥만 주면 품질과 비용이 같이 좋아진다.

가장 큰 비용

기준 문서와 대형 테스트 파일의 반복 탐색

spec, domain model, test plan, 큰 widget test를 매번 처음부터 훑는 순간 토큰이 크게 늘어난다.

운영 목표

AI가 읽는 입력량을 먼저 설계하기

구현 전에 컨텍스트 예산을 잡고, 실패 로그는 요약해서 보고, 추적 ID로 문서와 테스트를 연결한다.

COST MAP

어디서 토큰이 많이 쓰이는가

AI 개발툴은 코드를 쓰는 순간보다, 무엇을 지켜야 하는지 알아내는 순간에 토큰을 많이 쓴다. 아래 항목은 이 프로젝트의 SDD/TDD 구조에서 자주 발생하는 비용 순서다.

매우 큼 긴 기준 문서 반복 읽기

spec.md, feature spec, test_plan.md, domain_model.md를 작업마다 모두 열면 고정비가 커진다.

매우 큼 대형 테스트 파일 탐색

기존 패턴을 맞추려면 큰 widget/domain test를 보게 된다. 필요한 테스트명만 찾지 못하면 주변 코드까지 읽는다.

큼 동일 정책의 중복 검증

Firestore write boundary, ownership, 금지 copy가 여러 문서와 rules에 반복되면 일관성 확인 비용이 생긴다.

큼 긴 검증 로그

Flutter test 실패, analyzer 출력, build log는 실제 원인보다 주변 정보가 길다. 그대로 대화에 올리면 비용이 급증한다.

중간 불명확한 작업 범위

수정 범위가 흐리면 AI가 혹시 몰라 더 많이 읽는다. scope가 명확할수록 검색과 추론이 짧아진다.

OPERATING MODEL

문제는 SDD/TDD가 아니라 읽는 방식이다

SDD와 TDD는 AI에게 좋은 안전장치다. 다만 작업마다 전체 문서를 다 읽는 방식이면, 작은 변경도 큰 context load가 된다. 목표는 문서를 줄이는 게 아니라, 문서를 라우팅하는 것이다.

현재 자주 생기는 루프

정확하려고 많이 읽는 방식

AI가 불안해서 전체 기준을 훑고, 큰 테스트 파일을 열고, 실패 로그를 길게 해석한다.

METHOD

다음 작업부터 적용할 6단계 방법

핵심은 AI에게 일을 맡기기 전에 입력 문맥의 크기를 설계하는 것이다. 이 6단계는 SDD/TDD 순서를 깨지 않고 토큰 낭비만 줄인다.

1

작업을 먼저 분류한다

요청을 받으면 바로 파일을 열지 않고, 작업 유형을 먼저 고른다.

  • UI copy/layout
  • domain behavior
  • Firestore/rules
  • refactor
  • documentation-only
2

컨텍스트 예산을 잡는다

작업 시작 전 “처음 읽을 파일은 최대 4개”처럼 작은 예산을 둔다.

  • 작은 UI 수정: 2개에서 3개 파일
  • domain 변경: 4개에서 6개 파일
  • Firestore 변경: rules 포함 5개에서 7개 파일
3

문서에는 추적 ID를 붙인다

긴 test plan 항목을 사람이 읽는 문장 그대로 두되, 찾기 쉬운 ID를 붙인다.

  • Q-ANSWER-001: answer draft는 write하지 않음
  • MUSIC-COMMENT-002: comment save는 comment 문서만 씀
  • HOME-UNREAD-003: 새 소식은 3개만 preview
4

테스트명과 연결한다

각 추적 ID 아래에 실제 테스트 파일과 테스트명을 적는다.

  • AI가 큰 test file 전체를 읽지 않아도 된다.
  • 실패한 테스트가 어떤 spec을 지키는지 바로 보인다.
  • review 때 빠진 coverage를 발견하기 쉽다.
5

검증 로그를 잘라서 본다

긴 로그는 원문 전체를 대화에 올리지 않고, 실패 요약 형식으로 남긴다.

  • 실패 command
  • 첫 에러 1개에서 3개
  • 관련 파일과 라인
  • 재현 명령
6

handoff에 토큰 원인을 기록한다

작업이 끝나면 무엇을 많이 읽었는지 짧게 기록한다.

  • 많이 읽은 문서
  • 열지 않아도 됐던 파일
  • 다음부터 routing map에 추가할 단서
BEFORE AND AFTER

AI에게 이렇게 주면 비용이 달라진다

같은 기능 수정이라도 요청 방식에 따라 AI가 읽는 양이 크게 달라진다. 아래 예시는 “음악 댓글 저장 실패 retry”를 고친다고 가정한 것이다.

비싼 요청

전체 문맥을 다시 찾게 만드는 방식

요청 음악 댓글 저장 실패 처리를 고쳐줘. SDD/TDD 잘 맞게 해줘.
AI가 하게 되는 일 spec index부터 관련 문서 추정, test plan 전체 탐색, music/widget/domain/firestore 파일을 넓게 확인.
결과 정확성은 올라가지만, 탐색 토큰이 커지고 작업 초반이 길어진다.
절약되는 요청

작업별 컨텍스트 팩을 주는 방식

요청 MUSIC-COMMENT-RETRY 범위로 수정해줘. 읽을 문서는 spec/music.md, spec/domain_model.md, test_plan.md의 music comment 항목.
테스트 단서 alagagi_auth_test.dart에서 music note comment, retry, failed 검색. UI는 alagagi_app_test.dart의 music detail comment 테스트만 확인.
결과 AI가 어디를 봐야 하는지 바로 알기 때문에 SDD/TDD 품질은 유지되고 탐색 비용은 줄어든다.
TEMPLATE

추가하면 좋은 작은 문서 2개

새 거대한 문서를 만들자는 뜻이 아니다. AI가 시작할 때 방향을 잡는 얇은 지도와, 긴 test plan을 빠르게 찾게 해주는 색인만 추가한다.

MEASUREMENT

토큰을 줄였는지 확인하는 지표

실제 token count를 매번 정확히 재지 않아도 된다. 대신 작업마다 읽은 파일 수, 대형 파일 접근 횟수, 로그 길이, 재탐색 횟수를 보면 개선 여부가 보인다.

작업 handoff에 남길 작은 계측

이 네 가지를 기록하면 어떤 작업이 토큰을 많이 먹는지 금방 패턴이 보인다.

3개 이하 작은 UI/copy 수정에서 처음 읽는 파일 목표
6개 이하 domain behavior 변경에서 처음 읽는 파일 목표
30줄 이하 대화에 공유하는 실패 로그 요약 목표
1회 이하 같은 문서를 다시 넓게 훑는 재탐색 목표

실행 제안

1차 적용: 문서 추가만 한다

먼저 docs/ai_context_map.md와 docs/spec_trace.md를 얇게 만든다. production code, Firestore write, UI 동작은 바꾸지 않는다.

2차 적용: AGENTS.md에 짧게 연결한다

AI agent가 작업 시작 시 routing map을 먼저 보도록 한 줄을 추가한다. 기존 SDD/TDD 순서는 그대로 유지한다.

3차 적용: test_plan에 ID를 점진적으로 붙인다

한 번에 전체를 바꾸지 말고, 다음 변경이 닿는 항목부터 ID를 붙인다. 이렇게 하면 문서 정리 자체가 큰 비용이 되지 않는다.

4차 적용: 검증 로그 요약 규칙을 둔다

실패 로그는 원문 전체 대신 첫 에러, 관련 파일, 재현 명령만 handoff한다. full log는 로컬 터미널에 남기고 대화에는 요약만 올린다.
