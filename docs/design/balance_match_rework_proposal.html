“취향 매치”로 전면 개편하는 기안

지금의 밸런스 게임은 선택 이후의 보상이 약해서 사용 이유가 흐립니다. 전면 개편 방향은 단순 A/B 선택이 아니라, 서로의 선택을 열어보고 이유를 남기며 장소·만남·음악으로 자연스럽게 이어지는 작은 발견 경험으로 바꾸는 것입니다.

NAME

밸런스 게임 → 취향 매치

게임보다 덜 가볍고, 질문/소개카드와 다르게 “둘의 차이와 닮음”을 즉시 보여주는 이름입니다.

ROLE

대화 소재를 만드는 기능

정답을 맞히는 기능이 아니라, 서로의 선택 이유를 보고 다음 대화를 여는 기능으로 둡니다.

PAYOFF

결과 후 연결 행동

장소 후보, 만남 계획, 음악 노트로 이어져야 탭이 독립된 섬처럼 보이지 않습니다.

TONE

승패 없이 다정하게

같으면 “닮았어요”, 다르면 “다르게 좋아해요”처럼 평가 대신 발견의 문구를 씁니다.

DEVELOPMENT BRIEF

개발 기안

기존 밸런스 선택 저장 구조를 완전히 버리기보다, 선택/결과/연결 정보를 확장하는 쪽이 안전합니다. MVP는 “선택 + 이유 + 결과 공개 + 기록 필터”까지, 2차에서 장소/만남/음악 연결을 붙이는 흐름을 권장합니다.

핵심 기능 범위

취향 매치는 질문형 기능과 겹치지 않도록 “둘 중 하나를 고르고, 둘의 결과를 비교하는 기능”으로 명확히 분리합니다.

  • 매일 또는 주기적으로 추천되는 `오늘의 취향 카드`를 제공한다.
  • 사용자가 선택하기 전에는 상대 선택을 숨기고, 선택 후 결과를 공개한다.
  • 선택 이유는 선택 후 선택적으로 한 줄만 남길 수 있게 한다.
  • 결과는 같음/다름/아직 기다림 세 상태로 구분한다.
  • 장소형, 만남형, 음악형 질문은 결과 화면에서 관련 기능으로 이어지는 버튼을 제공한다.
  • 기록 화면은 전체, 같았던 선택, 달랐던 선택, 연결된 선택으로 필터링한다.

추천 데이터 모델

balancePrompts 질문 카탈로그. id, title, category, optionA, optionB, bridgeType, active 여부.
balanceSelections 사용자 선택. promptId, profileId, optionId, reason, createdAt, updatedAt.
balanceLinks 결과에서 연결한 기능. promptId, linkedType(place/meeting/music), linkedTargetId.
local seen state 새 매치/상대 선택 도착 여부는 홈 새 소식과 연결. 읽음 상태는 localStorage 우선.

개발 단계

MVP: 취향 매치 탭 리브랜딩 탭 이름/카피/화면 구조를 바꾸고, 기존 밸런스 선택 저장 로직을 새 카드 UI에 연결합니다.
결과 공개와 이유 한 줄 내 선택 저장 후 상대 선택이 있으면 결과를 공개하고, 선택 이유 필드를 추가합니다.
기록 필터와 새 소식 같음/다름/기다림 상태를 기준으로 기록을 필터링하고, 상대의 새 선택을 홈에서 알려줍니다.
다른 기능으로 연결 장소 후보 담기, 만남 계획 추가, 음악 노트 작성 진입을 결과 화면 액션으로 연결합니다.

검증 기준

  • 선택 전에는 상대 선택이 화면에 노출되지 않는다.
  • 내 선택 저장은 명시적 버튼 탭에서만 Firestore write를 만든다.
  • 상대 선택이 이미 있으면 결과 카드가 즉시 열린다.
  • 결과가 다른 기능으로 연결되어도 원본 선택 기록은 유지된다.
  • 질문 카피는 사귀는 관계를 전제하지 않고, 부담 없는 취향 발견 톤을 유지한다.
  • 모바일에서 A/B 카드 텍스트가 넘치지 않고 터치 영역이 충분하다.
